Summary of Project

The Homework asked to design a temperature input system for authorized users. This application, while raw in nature, achieves the required tasks by the following:

  1. The awaiting diagram is current values as sensor upper bound, lower bound, and current temperature. The values in the chart are hard coded. But, an assessor can go to the list data page to see inputted values.
  2. The sensors in this case are the upper and lower bounds. These are set by the user, if authorized. There is no authentication in the web app, other than the pseudocode that supports the front end capacity of this function. See index.html for the if user.is_authenticated function.
  3. Awareness is supported in two ways. First there is listed history of changes from the user. "List of Temp Changes." Second, there is chart data to display only current or most recent values. This model also executes the average of all bounded temperatures
  4. If the user is authorized to change the temperature, they can do so, but only within their specified bounds. "Change the Data" uses a form class to reevie this input. If the input is out of bounds, the record will not update.
  5. There is a Admin to the site, which can override a user. I.e., to send updates to the codebase or change raw data if needed. An attachment of this file is in the repo for this site. The link is at the top of this page.
  6. The user interface would be a combination of the form submission and the chart data.
  7. Exception Handling is handled by validation errors for bad input. If there is a bad value, a validation error is raised on the back end, and that input is not saved to the model data.

Common Design Problem Challenges

  1. Targets the Wrong Problem: This problem was easy to overcome in the fact that the requirements were clearly defined for this project. My minimal example is meant to mimic something like a Nest System would do. You could set the temp to 106! But staying within averages and reducing interactions with the device is what I imagine the problem to be from the requirements (i.e. maximize automation). However, that is a subjective opinion. Additionally, Nest generates a Home Report that shares your data with you each month, shows energy usage, etc. to make users aware of their over/under utilization.
  2. Leads to an inefficient solution: The solution in this case still requires input. So while, in theory, you are restricting the values through validation, there is still a more efficient solution in automating the whole process. Design challenges work in black and white but rarely address gray areas. The hope would be to oscillate around the average temperature and reduce costs. Ease of Use was something to consider here in which the UI/UX was as simple as possible.
  3. Does not differ significantly from other abstractions: Our solution uses an authorized web application instead of a hardware device to change the temperatures. It is fundamentally different in receiving its input. The idea is that the more data the user has, the more aware they are of their own behavior. The significant difference in this case is that there is a database to store and retrieve data. It can also log certain errors for its own corrective process. Cloud data could be used to benchmark homes/rooms.
